02
GET /api/v1/cart returns 400 Bad Request — missing x-session-id header
Bug API / Backend Auth / Session
Root Cause Guest cart functionality on the backend requires an x-session-id header to identify anonymous sessions. The frontend Axios instance had no logic to generate or attach this header. Additionally, CORS was configured without X-Session-ID in its allowedHeaders list, so the header was being stripped by CORS preflight even when manually sent.
Fix Applied In the Axios request interceptor inside api.ts, a UUID is now generated once with crypto.randomUUID(), stored in localStorage under the key 'session-id', and automatically attached as x-session-id on every outgoing request. 03
Runtime crash: product.averageRating.toFixed is not a function
Bug API / Backend
Root Cause Prisma stores numeric columns like averageRating and price using its internal Decimal type. When Prisma serializes a model to JSON (e.g. in an API response), Decimal fields are converted to a plain string rather than a JavaScript number. The frontend assumed averageRating was a number, so calling .toFixed(1) on a string threw a runtime TypeError.
Fix Applied Added a global override in the Prisma database config: Decimal.prototype.toJSON = function() { return parseFloat(this.toString()); } This ensures every Decimal field is serialized as a native JavaScript number when JSON.stringify is called, so all callers on the frontend can use numeric methods like .toFixed() safely.

backend/src/configs/database.ts
04
TanStack Query error: "Query data cannot be undefined" — affects 17 files
Bug API Layer
Root Cause The Axios instance in api.ts uses a response interceptor that already unwraps the standard API envelope { success, data, message, meta }, returning the inner data payload. All 17 affected files were still performing a second unwrap: api.get<ApiResponse<X>>(...).then(r => r.data.data). The double-unwrap caused r.data.data to be undefined since r.data was already the final payload — TanStack Query rejects undefined query data by design.
Fix Applied All 17 query functions were updated to the correct single-unwrap pattern: api.get<X>(...).then(r => r.data). Generic type parameters were updated from ApiResponse<X> to X throughout, removing the redundant wrapper type.

06
Category filter returns empty results when clicking a parent category
Bug API / Backend
Root Cause The product repository filtered by an exact categoryId match. The category hierarchy has parent and child (subcategory) nodes, and products are only ever assigned to leaf subcategories, not parent categories. Selecting a parent category produced zero results because no product directly carried that parent's ID.
Fix Applied Added a recursive helper function getCategoryDescendantIds(categoryId) to products.repository.ts. It walks the category tree downward and collects all descendant IDs. 07
isFeatured Zod transform silently applies false filter to all requests
Bug API / Backend
Root Cause The Zod schema for product query parameters defined: isFeatured: z.string().optional().transform(v => v === 'true'). When isFeatured was absent from the request (i.e., undefined), the transform evaluated undefined === 'true' which is false. Zod passed this false value through, and the database query added a WHERE isFeatured = false clause on every request that omitted the parameter — silently filtering out all featured products from regular listings. The same class of bug existed in numeric query parameters such as minPrice and maxPrice.
Fix Applied All boolean and numeric Zod transforms were updated to preserve undefined when the input is absent: transform(v => v !== undefined ? v === 'true' : undefined). This ensures that an omitted filter parameter results in undefined, which Prisma ignores entirely rather than filtering on.

backend/src/validators/product.validators.ts

09
Stripe Elements throws "invalid clientSecret" for mock payment intents
Bug UI / Checkout
Root Cause When Stripe is not configured in the environment, the backend returned a mock clientSecret formatted as pi_mock_xxx_secret_demo. Stripe.js strictly validates the clientSecret format before mounting <Elements>: it expects a string in the pattern pi_[id]_secret_[key] with exactly one underscore separating each segment. The extra underscores in pi_mock_xxx caused Stripe's internal regex to reject the secret before any network call was made.
Fix Applied The backend mock secret format was changed to mock_{orderId} (no pi_ prefix). In CheckoutForm.tsx, a guard was added: if clientSecret.startsWith('mock_') the component skips the Stripe <Elements> tree entirely and renders a "Demo Mode — Complete Order" button. Clicking this button calls POST /payments/confirm directly with a synthetic payment intent ID, bypassing Stripe altogether.

Frontend Architecture

📁 Project Structure

The frontend/src/ directory follows a features-first layout. Each business domain (auth, cart, checkout, orders, products, vendor, admin) owns its components, hooks, and related logic in features/. Cross-cutting concerns (routing, layouts, shared UI primitives, global stores, and the API layer) live in top-level sibling directories.

frontend/src/ ├── components/ │ ├── shared/ # Header, Footer, Sidebar, ProtectedRoute, RoleGuard │ └── ui/ # Button, Avatar, Dropdown, Skeleton, Spinner, Pagination … │ ├── features/ │ ├── auth/ # LoginForm, RegisterForm, hooks (useLogin, useRegister) │ ├── cart/ # CartDrawer, CartItem, useCart hook │ ├── checkout/ # CheckoutForm, AddressForm, PaymentForm, OrderSummary │ ├── orders/ # OrderList, OrderStatus badge, useOrders hook │ ├── products/ # ProductCard, ProductSearch, RelatedProducts, ProductReviews │ ├── vendor/ # VendorDashboard, VendorProducts, VendorOrders, VendorAnalytics, ProductForm │ └── admin/ # AdminDashboard, UserManagement, VendorApproval, ProductModeration │ ├── hooks/ # useAuth, usePagination, useDebounce ├── layouts/ # MainLayout, AuthLayout, DashboardLayout ├── pages/ # One file per route: Home, Products, ProductDetail, Cart, Checkout … ├── routes/ │ └── index.tsx # Complete route tree — TanStack Router code-based ├── services/ # API service modules per domain (order.service, vendor.service …) ├── store/ │ ├── auth.store.ts # Zustand + persist — user, tokens, isAuthenticated │ └── cart.store.ts # Zustand + persist — items, itemCount, subtotal, isOpen ├── types/ # TypeScript interfaces per domain └── utils/ ├── api.ts # Axios instance — interceptors, envelope unwrap ├── format.ts # Currency, date, number formatters ├── cn.ts # Tailwind className merge utility └── storage.ts # localStorage helpers
💡
Convention: Every feature directory is self-contained. A feature should not import from another feature's internals. Cross-feature communication happens through shared stores (store/), shared hooks (hooks/), or shared UI primitives (components/ui/).

🗺️ Routing Architecture

Routing is handled entirely by TanStack Router using a code-based route tree defined in routes/index.tsx. There are no file-based conventions. Three distinct layout branches exist; each branch applies different chrome (header, sidebar, auth card) and different access guards.

🌐 mainLayoutRoute

Wraps all public-facing and customer pages. Renders a sticky header, a centred container, and a footer. Used for: Home, Products, Product Detail, Cart, Checkout, Orders, Profile, static pages.

🔐 authLayoutRoute

Centered card layout — no header or footer. Used exclusively for: Login, Register, Forgot Password, Reset Password.

🏢 Vendor / Admin Sub-routes

Vendor and admin pages do not use a separate top-level layout route. Instead, the route component itself is wrapped in a role-aware gate:

  • VendorGate — Reads the user's role. Renders DashboardLayout (sidebar) for VENDOR/ADMIN; renders a plain "Become a Vendor" page for CUSTOMER.
  • VendorSubRoute — Strict guard; redirects non-vendors to /vendor/dashboard (which then lands on the gate).
  • AdminRoute — Strict guard; redirects non-admins to /.

Route guard wrappers — all three are React components wrapping <Outlet />:

🗃️ State Management

Global client state is managed by two Zustand stores, both configured with the persist middleware to survive page refreshes. Server state (products, orders, cart data from the API) is managed by TanStack Query.

🔐 auth.store — persisted to localStorage key 'auth-store'
user User | null Full user profile object (id, email, name, role, avatar)
accessToken string | null JWT access token injected into every API request via Authorization header
refreshToken string | null JWT refresh token used to obtain a new access token on 401
isAuthenticated boolean Derived from whether accessToken and user are non-null
setAuth() action Sets user + both tokens on login / token refresh
updateUser() action Partially updates the user object (used after /users/me refresh — Bug 18)
clearAuth() action Resets all fields to null/false on logout or 401
setAccessToken() action Updates only the access token after a silent refresh
🛒 cart.store — persisted to localStorage (isOpen excluded)
items CartItem[] Array of cart line items (productId, quantity, price, name, image)
itemCount number Total quantity of all items — displayed in the header badge
subtotal number Sum of item prices before discounts/shipping
total number Final amount including tax, shipping, coupon deductions
coupon Coupon | null Applied coupon code and discount amount if present
isOpen boolean Controls CartDrawer slide-over visibility — NOT persisted
setCart() action Syncs full cart data from the server response into the store
clearLocalCart() action Resets items/count/totals to zero — called on checkout success (Bug 10)
toggleCart() action Flips isOpen — called from cart icon in Header
ℹ️
Pattern: Zustand stores hold UI state and auth credentials. Actual server data (product lists, order history, cart contents from the API) is always fetched and cached by TanStack Query. The cart store is synced via setCart() inside the cart query's onSuccess callback, making the store a projection of server state rather than the source of truth.

🔌 API Layer — utils/api.ts

A single Axios instance is the only mechanism for all HTTP calls. It handles authentication, session tracking, response normalisation, and automatic logout on 401. No raw fetch() calls exist in the codebase. 

// Axios instance — simplified illustration
const api = axios.create({
  baseURL: import.meta.env.VITE_API_URL,
  headers: { 'Content-Type': 'application/json' },
});

// ── Request interceptor ──────────────────────────────────────
api.interceptors.request.use((config) => {
  // 1. Auth token
  const token = useAuthStore.getState().accessToken;
  if (token) config.headers.Authorization = `Bearer ${token}`;

  // 2. Guest session tracking (Bug #2 fix)
  let sessionId = localStorage.getItem('session-id');
  if (!sessionId) {
    sessionId = crypto.randomUUID();
    localStorage.setItem('session-id', sessionId);
  }
  config.headers['x-session-id'] = sessionId;
  return config;
});

// ── Response interceptor ─────────────────────────────────────
api.interceptors.response.use(
  (response) => {
    // Unwrap API envelope: { success, data, message, meta }
    // Returns the inner payload — callers use api.get<T>().then(r => r.data)
    return response.data;
  },
  (error) => {
    if (error.response?.status === 401) {
      // Trigger global logout — ProtectedRoute listens for this event
      window.dispatchEvent(new Event('auth:logout'));
    }
    return Promise.reject(error);
  }
);

💳 Checkout Flow

The checkout experience is a multi-step process orchestrated entirely within CheckoutForm.tsx. It progresses from address collection to payment and finally order confirmation, with a special Demo Mode path when Stripe is not configured.

Step 1
Address Form
User fills inline shipping address fields. handleAddressSubmit fires.
→
Step 2
POST /orders
Sends { shippingAddress, couponCode } → receives { orderId, clientSecret, amount }.
→
Step 3
Payment Gate
If clientSecret.startsWith('mock_') → Demo Mode button. Otherwise → Stripe Elements.
→
Step 4
stripe.confirmPayment()
Stripe.js handles card tokenisation. Returns paymentIntent.id on success.
→
Step 5
POST /payments/confirm
Sends { orderId, paymentIntentId }. Backend marks order CONFIRMED.
→
Step 6

// CheckoutForm — abridged confirm flow
const confirmMutation = useMutation({
  mutationFn: ({ orderId, paymentIntentId }) =>
    confirmOrder({ orderId, paymentIntentId }),

  onSuccess: () => {
    clearLocalCart();                              // Bug #10 fix
    queryClient.invalidateQueries({ queryKey: ['cart'] });
    navigate({ to: '/order-success' });
  },
});

// Demo Mode path (Bug #9 fix)
if (clientSecret.startsWith('mock_')) {
  return (
    <DemoModeButton
      onClick={() => confirmMutation.mutate({
        orderId,
        paymentIntentId: `demo_${Date.now()}`,
      })}
    />
  );

🔍 ProductSearch / Products Page — URL-driven Filter State

Filter state (search query, category, price range, sort) lives exclusively in the URL as search params, not in local component state. This makes every filtered view shareable via link and navigable via the browser back button.

  • Reading state: useSearch() from TanStack Router returns the current URL search params as a typed object.
  • Writing state: useNavigate() is called with the updated params object whenever a filter changes — no setState.
  • Data fetching: useQuery(['products', filters]) re-fires whenever filters (derived from URL params) changes.
  • Debouncing: The text search input passes through useDebounce(value, 400) before being written to the URL, preventing excessive requests on keystroke.
  • Pagination: page and limit are also URL params; the <Pagination> component writes them via useNavigate.
